Challenge 5 — IDOR: Insecure Direct Object Reference

You just booked a flight. AirNova sends you to this confirmation URL:

https://airnova.com/booking/confirmation?id=
Your mission — IDOR

The server returns booking data based on the id= parameter in the URL. It never checks if the logged-in user actually owns that booking.

Change the booking ID in the URL bar above and find booking #1038 — it belongs to another passenger and contains the flag.

In a real system this would expose real passenger names, passport numbers, payment info, and travel plans.

FLAG{1d0r_n0_4uthor1z4t10n_ch3ck} — You accessed another passenger's booking without authorization. This is IDOR (Insecure Direct Object Reference) — one of the most common and impactful web vulnerabilities. The fix: always verify server-side that the requesting user owns the resource.
